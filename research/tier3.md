# Tier 3 Scan

## drupal_ai

The link `https://www.drupal.org/project/drupal_ai` does not appear to be the canonical project slug; current Drupal.org search results resolve the main project as [`drupal/ai`](https://www.drupal.org/project/ai). That project is clearly **out of scope** for this research: it is the Drupal module ecosystem for adding AI capabilities to Drupal sites, with AI Core, AI Automators, AI Search, provider integrations, and site-building/editor workflows, not a skill library for people using AI to build Drupal software. The current project page says it was created on **March 7, 2023**, updated on **March 16, 2026**, and has stable releases including **1.3.2** on **April 1, 2026**. Treat it as adjacent ecosystem context, not a merge target. Sources: [Drupal AI project page](https://www.drupal.org/project/ai), [Drupal AI ecosystem docs](https://www.drupal.org/docs/develop/development-tools/ai-coding-tools-for-drupal-development/ai-tools-and-projects-in-the-drupal-ecosystem).

## drupal_devkit

`drupal_devkit` **does exist** on Drupal.org at [`drupal.org/project/drupal_devkit`](https://www.drupal.org/project/drupal_devkit). It is not another content-library repo in the same sense as `ai_best_practices`; it presents itself as a “marketplace of marketplaces” and cross-harness installer for AI skills, agents, and context files, with a Python CLI that installs skills across Claude Code, Codex, Cursor, Gemini, OpenCode, Kiro, and others. The project page explicitly positions `ai_best_practices` as a marketplace/content source that Devkit can install, which makes this complementary infrastructure rather than a competitor. For this research, treat it as an important adjacent delivery/distribution layer, closer in role to Surge than to a skill-content repo. Sources: [Drupal Devkit project page](https://www.drupal.org/project/drupal_devkit).

## awesome-drupal-ai

[`AJV009/awesome-drupal-ai`](https://github.com/AJV009/awesome-drupal-ai) is a broad, community-maintained **directory/index**, not a primary implementation target. Its README describes itself as “A curated list of AI tools, skills, plugins, modules, and resources for Drupal development and site building” and claims coverage across modules, skill repos, DDEV plugins, MCP servers, and standards. That makes it useful for discovery and gap-checking, but not something to merge into `ai_best_practices` except indirectly via links or periodic scanning. It is best treated as an index of the wider landscape, especially for spotting adjacent repos that fall outside the narrow Tier 1/Tier 2 set. Sources: [GitHub repository](https://github.com/AJV009/awesome-drupal-ai), [README.md](https://github.com/AJV009/awesome-drupal-ai/blob/main/README.md).

## Zivtech Landscape Page

The page at [`zivtech.github.io/zivtech-demos/drupal-ai-skills-landscape.html`](https://zivtech.github.io/zivtech-demos/drupal-ai-skills-landscape.html) appears to function as a **landscape/discovery aid**, not as a canonical source of truth. What is verifiable is that the `ai_best_practices` issue queue explicitly cites this page as the path by which issue [#3565489](https://www.drupal.org/project/ai/issues/3565489) was discovered, which supports treating it as historically useful index material rather than a repo or package to analyze as a merge candidate. Source: [ai_best_practices issue #3581836](https://www.drupal.org/project/ai_best_practices/issues/3581836).

## skills.sh?q=drupal

[`skills.sh`](https://skills.sh/) is an ecosystem-wide **directory and installer surface** for Agent Skills, not a Drupal-specific repo. The site presents itself as “The Open Agent Skills Ecosystem,” and the Drupal documentation page on AI coding tools explicitly points people to `skills.sh/?q=drupal` to browse installable Drupal skills following the open standard. This makes it useful as an index for discovery, naming-collision checks, and install ergonomics, but not as a source of authoritative Drupal guidance. In other words: good for scanning what exists, bad as a substitute for evaluating the underlying repos. Sources: [skills.sh](https://skills.sh/), [Drupal docs on AI tools and projects](https://www.drupal.org/docs/develop/development-tools/ai-coding-tools-for-drupal-development/ai-tools-and-projects-in-the-drupal-ecosystem).

## ai_skills (historical special case)

[`ai_skills`](https://www.drupal.org/project/ai_skills) is historically informative but now explicitly **unsupported and obsolete**. The project page says “We are rallying behind the shared efforts over at `ai_best_practices`,” stores skills in a root `skills` directory, validates them on merge requests, and lists only three included skills: `drupal-coding-standards`, `drupal-issue-queue`, and `drupal-contribute-fix`. Since those map directly to repos or skill packages already covered elsewhere in this research, `ai_skills` does not add much unique content beyond confirming early community direction: root `skills/`, validation, Surge compatibility, and agentskills.io alignment. Treat it as a historical waypoint and absorb useful content from the underlying skills, not from this package wrapper itself. Sources: [Agentic Skills project page](https://www.drupal.org/project/ai_skills).

## Proposal #3565489

Issue [#3565489: “Proposal: First-class support for agent-skills”](https://www.drupal.org/project/ai/issues/3565489) is not a repo, but it is architecturally important context. The issue frames the need for Drupal-wide, shareable, project-agnostic skills; argues for avoiding vendor lock-in; proposes a standard `skills` folder for modules and tools; and discusses Composer/scaffolding/aggregation patterns that later show up in projects like Surge, `ai_skills`, and now `ai_best_practices`. For this synthesis, it should be treated as one of the clearest upstream statements of the core debates about packaging, aggregation, module-level skill ownership, and multi-tool compatibility. Source: [Drupal AI issue #3565489](https://www.drupal.org/project/ai/issues/3565489).
